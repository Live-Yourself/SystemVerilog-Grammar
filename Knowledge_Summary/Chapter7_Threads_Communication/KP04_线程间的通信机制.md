## 4.1 event事件 - 线程间简单信号通知

### 基本概念

event是SystemVerilog中最简单的线程间通信机制，用于**一个线程向另一个线程发送信号通知**。event就像一个"事件标志"，可以触发（trigger）和等待（wait）。

**类比理解：**
```
event类似于现实生活中的"门铃":
- 按门铃 → 触发事件（->e）
- 等待门铃 → 等待事件（@e）
- 听到门铃 → 事件触发，继续执行
```

**event的特性：**
- 轻量级：开销最小，效率最高
- 单向通知：只能通知事件已发生，不能传递数据
- 即时性：事件触发后，等待的线程立即响应
- 一次性：需要重复触发（使用循环）

### event的三要素

| 要素 | 语法 | 说明 | 示例 |
|------|------|------|------|
| **声明** | `event e;` | 声明一个事件 | `event data_ready;` |
| **触发** | `->e;` | 触发事件 | `->data_ready;` |
| **等待** | `@e;` | 等待事件触发 | `@data_ready;` |

**基本流程：**
```systemverilog
event e;  // 1. 声明事件

fork
  // 线程1: 等待事件
  begin
    $display("等待事件...");
    @e;  // 3. 等待事件（阻塞）
    $display("事件触发，继续执行");
  end
  
  // 线程2: 触发事件
  begin
    #10;
    $display("触发事件");
    ->e;  // 2. 触发事件
  end
join
```

### event声明与使用

**声明方式：**
```systemverilog
event data_ready;
event transfer_done;
event e1, e2, e3;           // 多个声明
event events[10];           // 数组
```

**触发（->操作符）：**
```systemverilog
// 触发是非阻塞的：->e 立即返回
// 触发是即时的：等待的线程立即被唤醒
// 触发不传递数据：只通知事件已发生

initial begin
  #10;
  ->e;  // 在时间10触发事件
end
```

### 两种等待方式

#### @（边沿触发）

```systemverilog
@e;  // 必须在等待之后触发才有效
```

**注意：** 如果事件在`@e`之前已触发，`@e`会**无限等待**！

#### wait(e.triggered)（电平触发）

```systemverilog
wait(e.triggered);  // 即使之前触发，也能检测到
```

**推荐使用`wait(e.triggered)`，更安全，不会错过事件。**

### event合并

多个event可以使用 `or` 合并，等待**任意一个**event触发：

```systemverilog
event data_ready, timeout, error;

@(data_ready or timeout or error);

if (data_ready.triggered)
  $display("数据接收成功");
else if (timeout.triggered)
  $display("接收超时");
else if (error.triggered)
  $display("接收错误");
```

### event使用注意事项

1. **@的边沿触发特性**：如果事件在`@e`之前触发，会无限等待。推荐用`wait(e.triggered)`
2. **事件触发丢失**：event触发后如果没有线程等待，该触发会丢失。可用标志位辅助
3. **多线程等待同一事件**：事件触发会**唤醒所有**等待线程

## 4.2 semaphore旗语 - 资源访问控制

### 基本概念

semaphore（旗语）用于**控制对共享资源访问**，限制同时访问某个资源的线程数量。

**类比理解：**
```
semaphore类似于"停车位":
- 总共有N个停车位 → semaphore有N个钥匙
- 来一辆车，拿一把钥匙 → get()获取资源
- 走一辆车，归还钥匙 → put()释放资源
- 没有车位时，车辆等待 → 资源不足时，线程阻塞
```

**semaphore的特性：**
- 资源计数：维护可用资源数量
- 互斥访问：确保资源不会被过度使用
- 阻塞等待：资源不足时，线程自动阻塞
- 先进先出：等待队列按FIFO顺序唤醒

### 创建与初始化

```systemverilog
semaphore sem;

// 创建有3个资源的semaphore
sem = new(3);

// 创建互斥锁（1个资源）
sem = new(1);

// 创建无资源的semaphore（初始为0，需要先put才能get）
sem = new(0);
```

### 核心操作

#### get() - 获取资源（阻塞）

```systemverilog
sem.get();    // 获取1个资源
sem.get(N);   // 获取N个资源
```

资源可用时立即返回，否则阻塞等待直到有资源。

#### put() - 释放资源

```systemverilog
sem.put();    // 释放1个资源
sem.put(N);   // 释放N个资源
```

资源计数增加，如果有等待的线程则自动唤醒。

#### try_get() - 非阻塞获取

```systemverilog
if (sem.try_get()) begin
  // 获取成功（返回1）
end
else begin
  // 获取失败，资源不可用（返回0）
  // 不阻塞，可以执行其他操作
end
```

### 典型应用场景

#### 场景1: 共享资源池

```systemverilog
semaphore resource_pool;

resource_pool = new(3);  // 3个DMA通道

// 线程1
resource_pool.get();   // 获得资源
#20;                    // 使用资源
resource_pool.put();   // 释放资源

// 线程4（资源已被占满）
resource_pool.get();   // 阻塞等待！直到有线程释放
```

#### 场景2: 互斥锁（mutex）

```systemverilog
semaphore mutex = new(1);  // 只有1个资源 = 互斥锁
int shared_counter;

// 线程1
mutex.get();           // 上锁
shared_counter++;      // 临界区操作
mutex.put();           // 解锁

// 线程2
mutex.get();           // 等待线程1解锁
shared_counter++;      // 临界区操作
mutex.put();           // 解锁
```

#### 场景3: 限流控制

```systemverilog
semaphore rate_limiter = new(5);  // 限制同时最多5个操作

for (int i = 0; i < 10; i++) begin
  fork
    begin
      rate_limiter.get();  // 超过5个时阻塞
      // ... 执行任务 ...
      rate_limiter.put();  // 释放
    end
  join_none
end
```

### semaphore使用注意事项

1. **必须成对使用get()和put()**：获取资源后不释放会导致资源泄漏
2. **资源计数不能为负**：semaphore内部计数不会小于0，get()会阻塞
3. **get(N)和put(N)是原子操作**：要么全部成功，要么全部失败

## 4.3 mailbox信箱 - 线程间数据传递

### 基本概念

mailbox（信箱）用于**线程间传递数据**，自带FIFO队列，是验证平台中组件间通信的核心机制。

**类比理解：**
```
mailbox类似于"邮局信箱":
- 写信 → 放入信箱 → put()发送数据
- 收信 → 从信箱取出 → get()接收数据
- 信箱容量 → 最多存放多少封信
- 信箱满 → 无法放入（阻塞）
- 信箱空 → 无法取出（阻塞）
```

**mailbox的特性：**
- 数据传递：可以传递任意类型的数据
- FIFO队列：先进先出的消息队列
- 容量限制：可以设置最大容量（或无限制）
- 阻塞操作：put()在满时阻塞，get()在空时阻塞
- 类型安全：可以指定传递的数据类型

### 创建与初始化

```systemverilog
// 无类型mailbox（可以传递任意类型）
mailbox mbx;

// 有类型mailbox（只能传递指定类型）
mailbox #(int) int_mb;
mailbox #(transaction) tx_mb;

// 创建无界信箱（容量无限）
mb = new();

// 创建有界信箱（容量为N）
mb = new(10);  // 最多存放10个消息

// 创建容量为0的信箱（必须有get()才能put()）
mb = new(0);  // 用于严格的同步
```

### 核心操作

#### put() - 发送数据（阻塞）

```systemverilog
mb.put(data);  // 信箱未满时立即返回，已满时阻塞等待
```

#### get() - 接收数据（阻塞）

```systemverilog
mb.get(data);  // 信箱非空时取出并移除，为空时阻塞等待
```

#### try_put() - 非阻塞发送

```systemverilog
if (mb.try_put(data))
  // 发送成功（信箱未满）
else
  // 发送失败（信箱已满），不阻塞
```

#### try_get() - 非阻塞接收

```systemverilog
if (mb.try_get(data))
  // 接收成功（信箱非空）
else
  // 接收失败（信箱为空），不阻塞
```

#### peek() - 查看数据（不移除）

```systemverilog
mb.peek(data);  // 查看但不移除，多次peek看到同一数据
```

#### num() - 查询消息数量

```systemverilog
int count = mb.num();  // 返回当前消息数量
```

### 有界mailbox vs 无界mailbox

| 特性 | 无界 `new()` | 有界 `new(N)` |
|------|-------------|--------------|
| 容量 | 无限 | 有限 |
| put()阻塞 | 永远不阻塞 | 容量满时阻塞 |
| 内存 | 可能消耗大量内存 | 内存可控 |
| 反压 | 无 | 有（自然反压） |

**建议：** 默认使用无界mailbox，除非有明确的容量限制需求。

### mailbox在验证中的典型应用

#### 生产者-消费者模型
```systemverilog
mailbox #(transaction) tx_mb = new();

// 生产者
tx_mb.put(tx);

// 消费者
tx_mb.get(tx);
```

#### 驱动器-记分板通信
```systemverilog
mailbox #(transaction) drv2sb_mb, mon2sb_mb;

// Driver发送
drv2sb_mb.put(tx);

// Scoreboard比较
drv2sb_mb.get(tx1);  // 驱动器事务
mon2sb_mb.get(tx2);  // 监视器事务
if (tx1.compare(tx2))  // 比较
```

### mailbox使用注意事项

1. **有界mailbox可能导致死锁**：所有线程都因信箱满/空而等待时
2. **类型必须匹配**：有类型mailbox只能传递指定类型
3. **忘记接收导致内存泄漏**：生产者持续put()，没有消费者get()
4. **peek()不会移除数据**：需要get()才能真正取出

## 4.4 三种通信机制对比总结

| 特性 | event | semaphore | mailbox |
|------|-------|-----------|---------|
| **核心概念** | 事件通知 | 资源控制 | 数据传递 |
| **声明方式** | `event e;` | `semaphore s;` | `mailbox #(type) m;` |
| **创建方式** | 无需创建 | `new(N)` | `new([size])` |
| **主要操作** | `->e`, `@e` | `get()`, `put()` | `put()`, `get()` |
| **传递数据** | 否 | 否 | 是 |
| **阻塞条件** | 等待事件 | 资源不足 | 信箱满/空 |
| **队列机制** | 无 | 无 | FIFO |
| **适用场景** | 状态通知 | 互斥访问 | 消息传递 |
| **性能开销** | 最低 | 低 | 较高 |

### 选择指南

```
需要线程通信吗？
  ├── 否 → 顺序执行
  └── 是 → 继续判断

需要传递数据吗？
  ├── 是 → 使用 mailbox
  └── 否 → 继续判断

需要控制资源访问吗？
  ├── 是 → 使用 semaphore
  └── 否 → 使用 event
```

---

**模块4总结：**

SystemVerilog提供了三种线程间通信机制：

1. **event** - 最轻量级的通知机制，适合只需要信号的场景。使用`wait(e.triggered)`比`@e`更安全
2. **semaphore** - 控制资源访问，确保线程安全。必须成对使用get()/put()，避免资源泄漏
3. **mailbox** - 传递数据的核心通道，自带FIFO队列。是验证平台组件通信的主要方式

验证平台中的经验法则：只需要通知用event，需要互斥用semaphore，需要传数据用mailbox。
