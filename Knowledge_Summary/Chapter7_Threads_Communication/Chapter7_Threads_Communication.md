## 模块1：线程的基本概念

### 1.1 什么是线程

在SystemVerilog中，线程（thread）也称为进程（process），是程序执行的基本单元。线程代表了**独立执行的代码序列**，它们可以并发运行，模拟硬件的并行行为。

**关键特性：**
- 每个线程有自己的执行路径和程序计数器
- 线程之间可以并行执行，互不干扰
- 线程可以通过同步和通信机制进行协调

### 1.2 线程与硬件并发的关系

```
硬件世界:
┌─────────────────────────────────────┐
│  寄存器1 ← 逻辑 → 寄存器2            │
│      ↓            ↓                  │
│  组合逻辑      组合逻辑              │
│      ↓            ↓                  │
│  寄存器3 ← 逻辑 → 寄存器4            │
└─────────────────────────────────────┘
       ↓              ↓
       并行执行      并行执行

软件模拟:
┌─────────────────────────────────────┐
│  线程1: 模拟寄存器1的行为            │
│  线程2: 模拟寄存器2的行为            │
│  线程3: 监控总线信号                 │
│  线程4: 生成时钟信号                 │
└─────────────────────────────────────┘
       ↓              ↓
       并发执行      并发执行
```

**类比理解：**
- **线程** 类似于硬件中的多个模块同时工作
- **线程并发** 模拟了硬件的并行性
- **线程同步** 模拟了硬件的握手协议和时序控制

### 1.3 验证环境中的并发需求

在验证平台中，为什么需要线程？

**典型场景：**

**场景1：多接口同时驱动**
```
DUT有3个输入接口，需要同时发送激励
线程1: 驱动接口A
线程2: 驱动接口B  
线程3: 驱动接口C
→ 模拟真实硬件环境中的并行输入
```

**场景2：监控与驱动并行**
```
线程1: 持续监控DUT输出
线程2: 向DUT发送激励
→ 边驱动边检查，提高验证效率
```

**场景3：超时检测机制**
```
线程1: 执行主要操作
线程2: 计时器，超时后报警
→ 防止测试挂死，提高可靠性
```

**场景4：协议握手**
```
线程1: 发送请求信号
线程2: 等待应答信号
线程3: 监控超时
→ 模拟复杂的协议交互
```

**并发需求总结表：**

| 需求场景 | 说明 | 线程数量 | 典型应用 |
|---------|------|---------|---------|
| 多接口驱动 | 同时驱动多个DUT接口 | 2-8个 | 多通道数据传输 |
| 监控与检查 | 持续监控DUT行为 | 1-3个 | 协议检查、覆盖率收集 |
| 超时管理 | 防止测试挂死 | 1个/操作 | 超时保护 |
| 时钟生成 | 产生时钟信号 | 1个/时钟 | 时钟、复位信号 |
| 数据收集 | 并行采集数据 | 多个 | 性能分析、调试信息 |

---

**模块1总结：**

线程是SystemVerilog验证中的核心概念，它使得我们可以：
1. **模拟硬件并行性** - 多个线程并发执行，模拟真实硬件行为
2. **提高验证效率** - 多个操作同时进行，缩短仿真时间
3. **增强可控性** - 精确控制各个操作的执行时机和顺序
4. **实现复杂场景** - 支持多协议、多接口的复杂交互验证

在验证平台中，合理使用线程是编写高效、可靠测试平台的关键。
