## 3.1 wait fork - 等待所有fork线程完成

### 基本概念

`wait fork` 用于**等待所有由当前线程启动的fork线程全部完成**。它通常用在 `fork...join_any` 或 `fork...join_none` 之后，确保所有后台线程都已完成。

**语法：**
```systemverilog
fork
  // 并行线程...
join_any  // 或 join_none

// 其他代码...

wait fork;  // 等待所有fork线程完成
```

### 适用场景

| 场景 | 说明 | 示例 |
|------|------|------|
| join_any后等待 | join_any后继续执行，但需要最终等待所有完成 | 超时检测后等待所有任务结束 |
| join_none后等待 | join_none启动后台任务，最后需要同步 | 后台初始化后等待完成 |
| 复杂流程控制 | 多阶段测试，每阶段有并行任务 | 阶段1并行任务→阶段2并行任务 |

### 执行流程

```
主线程
  │
  │     fork
  ├──────┬─────────────────────────────────────┐
  │      │                                     │
  │      │   线程1 ────────────────┐          │
  │      │                         │          │
  │      │   线程2 ───────┐        │          │
  │      │                │        │          │
  │      │   线程3 ───────┼────────┼──────┐   │
  │      │                │        │      │   │
  │      └────────────────┴────────┴──────┘   │
  │                     │        │      │     │
  │                     ▼        │      │     │
  │              ◄── join_any ──►│      │     │
  │                              │      │     │
  └──────────────────────────────┴──────┴─────┘
                                 │      │
                                 │      ▼
                                 │    其他代码
                                 │      │
                                 │      ▼
                                 │  wait fork
                                 │      │
                                 │      ▼
                                 │  等待线程2、3完成
                                 │      │
                                 │      ▼
                                 └────► 继续执行

说明：
- join_any在第一个线程完成时解除阻塞（线程2最快）
- 主线程继续执行其他代码
- wait fork等待所有后台线程（线程1、3）完成
- 最终所有线程都完成后再继续
```

### 典型应用

#### 应用1: join_any后等待所有完成
```systemverilog
fork
  begin : task_A
    #10;
    $display("任务A完成");
  end
  
  begin : task_B
    #5;
    $display("任务B完成");
  end
  
  begin : task_C
    #15;
    $display("任务C完成");
  end
join_any

$display("第一个任务完成，继续执行...");
// 执行其他操作...

wait fork;  // 等待剩余任务完成
$display("所有任务都已完成");

// 输出顺序:
// 任务B完成 (5ns)
// 第一个任务完成，继续执行... (5ns)
// 任务A完成 (10ns)
// 任务C完成 (15ns)
// 所有任务都已完成 (15ns)
```

#### 应用2: 多阶段并行任务
```systemverilog
// 阶段1: 并行配置
fork
  configure_module_A();
  configure_module_B();
  configure_module_C();
join_any

// 阶段1完成后的处理
setup_interrupt_handler();

// 等待阶段1所有配置完成
wait fork;

// 阶段2: 并行启动
fork
  start_module_A();
  start_module_B();
  start_module_C();
join

$display("所有模块启动完成");
```

## 3.2 disable fork - 终止所有fork线程

### 基本概念

`disable fork` 用于**立即终止所有由当前线程启动的、仍在运行的fork线程**。它常用于清理后台线程，或在发生错误时终止所有并行操作。

**语法：**
```systemverilog
fork
  // 并行线程...
join_any  // 或 join_none

disable fork;  // 终止所有后台线程
```

### 适用场景

| 场景 | 说明 | 示例 |
|------|------|------|
| 超时清理 | 超时后终止仍在运行的任务 | 超时检测后清理 |
| 错误处理 | 发生错误时终止所有并行操作 | 错误恢复 |
| 资源清理 | 测试结束时清理后台监控 | 测试退出清理 |
| 条件竞争 | 多个任务竞争，一个成功后终止其他 | 资源竞争 |

### 执行流程

```
主线程
  │
  │     fork
  ├──────┬─────────────────────────────────────┐
  │      │                                     │
  │      │   线程1 ────────────────┐          │
  │      │                         │          │
  │      │   线程2 ───────┐        │          │
  │      │                │        │          │
  │      │   线程3 ───────┼────────┼──────┐   │
  │      │                │        │      │   │
  │      └────────────────┴────────┴──────┘   │
  │                     │        │      │     │
  │                     ▼        │      │     │
  │              ◄── join_any ──►│      │     │
  │                              │      │     │
  └──────────────────────────────┴──────┴─────┘
                                 │      │
                                 │      ▼
                                 │  disable fork
                                 │      │
                                 │      ▼
                                 │  立即终止线程2、3
                                 │      │
                                 │      ▼
                                 └────► 继续执行

说明：
- join_any在第一个线程完成时解除阻塞（线程2最快）
- disable fork立即终止所有仍在运行的后台线程（线程1、3）
- 被终止的线程不会继续执行
- 主线程继续执行后续代码
```

### 典型应用

#### 应用1: 超时后清理
```systemverilog
bit success;

fork
  begin
    #actual_delay;
    success = 1;
  end
  
  begin
    #50;
    $display("超时！");
  end
join_any

if (success)
  $display("操作成功");
else begin
  $display("操作超时");
  disable fork;  // 终止仍在运行的任务
end
```

#### 应用2: 资源竞争（任一成功即终止其他）
```systemverilog
bit resource_A_done, resource_B_done;

fork
  begin
    #delay_A;
    resource_A_done = 1;
  end
  
  begin
    #delay_B;
    resource_B_done = 1;
  end
join_any

if (resource_A_done) begin
  $display("资源A就绪");
end
else if (resource_B_done) begin
  $display("资源B就绪");
end

disable fork;  // 终止仍在尝试的另一个资源请求
```

## 3.3 disable 命名块 - 终止特定线程

### 基本概念

`disable 命名块` 用于**终止特定的命名线程**。通过为begin块命名，可以精确控制要终止的线程，而不是终止所有线程。

**语法：**
```systemverilog
fork
  begin : block_A
    // 线程A的代码
  end
  
  begin : block_B
    // 线程B的代码
  end
join_none

disable block_A;  // 只终止block_A，block_B继续运行
```

### 适用场景

| 场景 | 说明 | 示例 |
|------|------|------|
| 选择性终止 | 只终止特定的线程 | 停止特定监控器 |
| 分级控制 | 对不同优先级的线程分别控制 | 高/低优先级任务 |
| 动态管理 | 根据条件动态决定终止哪个线程 | 条件判断 |

### 执行流程

```
主线程
  │
  │     fork
  ├──────┬─────────────────────────────────────┐
  │      │                                     │
  │      │   block_A ────────────────┐        │
  │      │   (命名块)                │        │
  │      │                         │        │
  │      │   block_B ───────┐      │        │
  │      │   (命名块)       │      │        │
  │      │                │      │        │
  │      └────────────────┴──────┴────────┘
  │                     │      │
  │                     ▼      │
  │              某些条件触发  │
  │                     │      │
  │                     ▼      │
  │              disable block_A
  │                     │      │
  │                     ▼      │
  │              立即终止block_A │
  │                     │      │
  │                     ▼      │
  │                 block_B继续运行
  │                     │      │
  │                     ▼      ▼
  └────────────────────► 继续执行

说明：
- block_A和block_B在后台运行
- 触发条件后，执行disable block_A
- block_A立即终止
- block_B不受影响，继续运行
```

### 典型应用

#### 应用1: 停止特定监控器
```systemverilog
fork
  begin : data_monitor
    forever begin
      @(data_bus);
      $display("数据总线变化: %h", data_bus);
    end
  end
  
  begin : timeout_monitor
    #100;
    $display("超时警告！");
  end
join_none

// 一段时间后停止数据监控，保留超时监控
#50;
disable data_monitor;  // 只停止数据监控

// 超时监控继续运行，直到100ns
```

#### 应用2: 条件性线程终止
```systemverilog
fork
  begin : fast_path
    #5;
    $display("快速路径完成");
  end
  
  begin : slow_path
    #30;
    $display("慢速路径完成");
  end
join_any

if (fast_path_done) begin
  $display("使用快速路径结果");
  disable slow_path;  // 终止慢速路径
end
```

## 3.4 线程控制的最佳实践

### 实践1: 始终管理后台线程

**不好的做法：**
```systemverilog
// 启动后台线程但不管理
fork
  forever begin
    // 监控任务...
  end
join_none

// 直接结束，后台线程继续运行
// 可能导致仿真挂死或资源泄漏
```

**好的做法：**
```systemverilog
// 启动后台线程并管理
fork
  begin : bg_monitor
    forever begin
      // 监控任务...
    end
  end
join_none

// 执行主任务...

// 清理后台线程
disable bg_monitor;  // 或 disable fork
```

### 实践2: 使用命名块提高可控性

**不好的做法：**
```systemverilog
fork
  #10;  // 匿名块
  
  #20;  // 匿名块
join_none

// 无法单独控制特定线程
disable fork;  // 只能全部终止
```

**好的做法：**
```systemverilog
fork
  begin : init_task
    #10;
  end
  
  begin : monitor_task
    #20;
  end
join_none

// 可以精确控制
disable init_task;  // 只终止init_task
// monitor_task继续运行
```

### 实践3: 合理使用wait fork和disable fork

**场景1: 需要最终同步** → 使用 `wait fork`
```systemverilog
fork
  task1();
  task2();
join_any

// 继续执行...

wait fork;  // 确保所有任务最终完成
```

**场景2: 需要立即清理** → 使用 `disable fork`
```systemverilog
fork
  task1();
  task2();
join_any

// 不需要后台任务继续
disable fork;  // 立即清理
```

### 实践4: 结合使用wait fork和disable fork

**复杂场景管理：**
```systemverilog
fork
  begin : critical_task
    // 关键任务
  end
  
  begin : optional_task1
    // 可选任务1
  end
  
  begin : optional_task2
    // 可选任务2
  end
join_any

if (critical_task_done) begin
  // 关键任务完成
  disable optional_task1;  // 终止可选任务
  disable optional_task2;  // 终止可选任务
end
else begin
  // 使用可选任务结果
  wait fork;  // 等待所有完成
end
```

## 3.5 线程控制对比总结

### wait fork vs disable fork

| 特性 | wait fork | disable fork |
|------|-----------|--------------|
| **作用** | 等待所有线程完成 | 立即终止所有线程 |
| **执行时机** | 被动等待 | 主动终止 |
| **线程状态** | 线程正常结束 | 线程被强制终止 |
| **后续代码** | 所有线程结束后继续 | 立即继续 |
| **适用场景** | 需要最终同步 | 需要立即清理 |

### 三种控制方式对比

| 控制方式 | 作用范围 | 精确度 | 使用场景 |
|---------|---------|--------|---------|
| **wait fork** | 所有fork线程 | 粗粒度 | 最终同步 |
| **disable fork** | 所有fork线程 | 粗粒度 | 全部清理 |
| **disable 命名块** | 特定命名线程 | 细粒度 | 精确控制 |

### 选择指南

```
需要管理后台线程吗？
  ├── 否 → 无需控制
  └── 是 → 继续判断

需要精确控制吗？
  ├── 是 → disable 命名块
  └── 否 → 继续判断

需要等待完成吗？
  ├── 是 → wait fork
  └── 否 → disable fork
```

**模块3总结：**

线程控制是验证平台中的重要技术：

1. **wait fork** - 等待所有fork线程完成，用于最终同步
2. **disable fork** - 终止所有fork线程，用于立即清理
3. **disable 命名块** - 终止特定线程，用于精确控制
4. **最佳实践** - 始终管理后台线程，使用命名块提高可控性

合理使用线程控制，可以编写出更健壮、可控性更好的验证平台！
