## 5.1 生产者-消费者模型

### 概述

生产者-消费者（Producer-Consumer）是验证平台中最常见的并发设计模式。生产者线程生成数据（事务），消费者线程接收并处理数据。两者通过通信机制解耦，各自独立运行。

### 核心思想

```
┌──────────┐     ┌───────────┐     ┌──────────┐
│  生产者   │ ──► │ 通信通道   │ ──► │  消费者   │
│(Producer) │     │(mailbox/  │     │(Consumer)│
│ 生成事务   │     │ semaphore)│     │ 处理事务  │
└──────────┘     └───────────┘     └──────────┘
```

**设计原则：**
- 生产者和消费者通过mailbox连接，解耦执行速度
- 生产者速度可以快于消费者（mailbox缓冲）
- 消费者速度也可以快于生产者（mailbox阻塞等待）
- 使用semaphore控制共享资源的访问

### 模型分类

| 类型 | 结构 | 适用场景 |
|------|------|---------|
| 单生产者-单消费者（1P1C） | Generator → mailbox → Driver | 单接口驱动、简单测试 |
| 多生产者-单消费者（NP1C） | Gen_A ┐ → mailbox → Driver | 多输入源合并、多master接口 |
| 单生产者-多消费者（1NPC） | Generator → mailbox → Driver_A/B/C | 广播模式、多组件接收 |
| 多生产者-多消费者（NPNPC） | Gen_A ┐ → merge → Driver_A/B | 复杂验证平台、多通道系统 |

### 关键实现要点

**1. 使用mailbox做数据通道：** `put()`放入、`get()`取出，FIFO顺序保证，自动阻塞/唤醒

**2. 事务对象的传递：** 传递的是对象句柄（引用），不是拷贝，效率高

**3. 终止机制：**
- 固定数量：生产者发送N个事务后停止
- 事件通知：通过event通知消费者结束
- 特殊标记：放入一个null或特殊事务作为结束标记

**4. 速率控制：** 使用有界mailbox（`new(size)`），容量满时`put()`自动阻塞，形成反压

### 典型验证平台应用

| 组件角色 | 生产者 | 消费者 | 通信机制 |
|---------|--------|--------|---------|
| 激励生成 | Generator | Sequencer | TLM端口/mailbox |
| 数据驱动 | Sequencer | Driver | TLM端口/mailbox |
| 响应收集 | Monitor | Scoreboard | Analysis端口/mailbox |
| 结果比较 | Ref Model | Scoreboard | TLM端口/mailbox |

## 5.2 握手协议实现

### 概述

握手协议（Handshake Protocol）是硬件接口间最常见的通信方式。在验证平台中，需要用线程和同步机制来精确模拟这些协议行为。

### 两阶段握手（Ready-Valid）

最基本的握手方式，两个信号协调数据传输。

```
Master                        Slave
│  valid  │ ────────────────► │
│         │ ◄──────────────── │  ready  │
│  data   │ ────────────────► │

传输条件：valid && ready 同时为高
```

**用event模拟：**
```systemverilog
event valid_event, ready_event;

// Master
->valid_event;
wait(ready_event.triggered);

// Slave
wait(valid_event.triggered);
#delay;
->ready_event;
```

### 三阶段握手（REQ-ACK）

更严格的握手，包含明确的请求和应答阶段。

```
Master                        Slave
│──── REQ ────►               │  阶段1: 请求
│◄── ACK ──────               │  阶段2: 应答
│──── REQ释放 ──►             │  阶段3: 完成

状态机: IDLE → REQ_SENT → ACK_RCVD → IDLE
```

**用mailbox模拟（带数据传递）：**
```systemverilog
mailbox #(transaction) req_mb;   // 请求信箱
mailbox #(response)   rsp_mb;   // 应答信箱

// Master
req_mb.put(tx);      // 发送请求（带数据）
rsp_mb.get(rsp);     // 等待应答（带数据）
```

### 关键设计要点

**1. 超时保护（必须）：** 任何握手都必须有超时机制，防止死锁
```systemverilog
fork
  wait(ack_event.triggered);
  handshake_complete = 1;
begin
  #100;
  $display("握手超时！");
end
join_any
if (!handshake_complete) disable fork;
```

**2. 多次握手循环：** 实际协议通常是多次握手，用for循环包裹单次握手

**3. 握手与数据分离：** 数据通道(mailbox)和同步通道(event)分开管理

### 常见握手模式

| 模式 | 信号 | 特点 | 适用协议 |
|------|------|------|---------|
| Ready-Valid | valid + ready | 无需应答阶段 | AXI、TileLink |
| REQ-ACK | req + ack | 有明确应答 | APB、Wishbone |
| Enable-Strobe | en + strobe | 简单触发 | 自定义接口 |
| IRQ-ACK | irq + ack | 中断确认 | 中断控制器 |

## 5.3 验证平台典型架构

### 概述

将前面学到的线程创建（fork...join系列）、线程控制（wait/disable fork）和通信机制（event/semaphore/mailbox）综合运用，构建一个典型的验证平台骨架。

### 典型验证平台架构

```
                    ┌─────────────────────┐
                    │      Generator       │
                    │   (生成事务)          │
                    └──────────┬──────────┘
                               │ mailbox
                    ┌──────────▼──────────┐
                    │      Driver          │
                    │   (驱动DUT)          │
                    └──────────┬──────────┘
                               │ 信号
              ┌────────────────┼────────────────┐
              │                │                │
       ┌──────▼──────┐ ┌──────▼──────┐ ┌──────▼──────┐
       │    DUT       │ │   Monitor   │ │  Ref Model  │
       │ (被测设计)   │ │  (采集响应) │ │  (参考模型)  │
       └─────────────┘ └──────┬──────┘ └──────┬──────┘
                               │ mailbox        │ mailbox
                    ┌──────────▼────────────────▼──────────┐
                    │          Scoreboard                    │
                    │        (比较结果)                      │
                    └───────────────────────────────────────┘
```

### 各组件的线程与通信需求

| 组件 | 线程机制 | 通信机制 |
|------|---------|---------|
| Generator | `fork...join_none`后台启动 | `mailbox.put()`发送事务 |
| Driver | `forever`循环 + `fork...join_any`(超时) | `mailbox.get()`接收 + `event`同步 |
| Monitor | `forever` + `fork...join_none` | `mailbox.put()`发送采集结果 |
| RefModel | `forever` + `mailbox.peek()` | `mailbox.put()`发送期望值 |
| Scoreboard | `fork...join_any`(三路竞争) | 两个`mailbox.get()` |

### 线程启动策略

使用 `fork...join_none` 启动各组件，使它们并行运行：

```systemverilog
fork run_generator(5); join_none
fork run_driver();      join_none
fork run_monitor();     join_none
fork run_ref_model();   join_none
fork run_scoreboard();  join_none
```

### 终止与清理策略

**步骤1：** Generator完成 → 触发`test_done`事件
**步骤2：** 等待pipeline排空 → `wait(test_done.triggered); #DRAIN_TIME;`
**步骤3：** 清理所有线程 → `disable fork;`
**步骤4：** 报告结果 → `$display("PASS/FALL"); $finish;`

### 综合运用总结

| 前面所学 | 在验证平台中的用途 |
|---------|------------------|
| fork...join | 等待所有组件初始化完成 |
| fork...join_none | 后台启动各组件线程 |
| fork...join_any | 超时检测 + 正常流程竞争 |
| wait fork | 等待所有后台线程完成 |
| disable fork | 仿真结束时清理 |
| event | 测试完成通知、握手同步 |
| semaphore | 共享资源访问保护（如寄存器配置） |
| mailbox | 组件间数据传递的核心通道 |

**验证平台的本质：多个线程通过mailbox通信，通过event同步，通过semaphore保护资源，共同协作完成验证任务。**

**模块5总结：**

综合应用是第7章的最终目标：

1. **生产者-消费者**是验证平台的基础设计模式，mailbox天然适配
2. **握手协议**用event模拟信号通知，用mailbox传递数据，必须带超时保护
3. **验证平台架构**将所有知识串联：fork创建线程、mailbox传递数据、event同步、disable清理
4. **"启动→等待→清理"**模式是验证平台的骨架
