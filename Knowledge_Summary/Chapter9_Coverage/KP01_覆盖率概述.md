# KP01 覆盖率概述

## 核心概念

覆盖率（Coverage）是衡量验证充分程度的量化指标。在 SystemVerilog 中，覆盖率分为两大类：

| 类型 | 收集方式 | 关注点 |
|:----:|:--------:|--------|
| **代码覆盖率** (Code Coverage) | 仿真器自动收集 | 哪些代码行/分支/条件被执行过 |
| **功能覆盖率** (Functional Coverage) | 验证工程师主动定义 | 设计规格中的功能点是否都验证到了 |

## 1. 代码覆盖率

仿真器在仿真过程中自动统计，无需额外编写代码。常见的代码覆盖率类型：

- **行覆盖率（Line Coverage）**：源代码的每一行是否被执行过
- **翻转覆盖率（Toggle Coverage）**：每个比特位是否经历了 0→1 和 1→0 的翻转
- **条件覆盖率（Condition Coverage）**：if/case 中的每个分支是否都被走到
- **表达式覆盖率（Expression Coverage）**：布尔表达式的各种取值组合是否都被覆盖

代码覆盖率的问题是：**代码被执行过不等于功能正确**，也无法告诉你"是否所有该验证的场景都验证了"。

## 2. 功能覆盖率

功能覆盖率由验证工程师根据**设计规格**主动定义，它回答的是："我关心的功能点是否都被验证到了？"

实现方式：通过 `covergroup` 定义覆盖率模型，在仿真过程中调用 `sample()` 对数据进行采样，仿真结束后查看覆盖率报告。

## 3. 覆盖率驱动验证的基本思想

```
定义覆盖率 → 运行仿真 → 查看覆盖率报告 → 发现缺口 → 调整激励 → 循环
```

验证工程师不再盲目地写测试用例，而是以覆盖率为目标，有针对性地生成激励。当功能覆盖率达标时，意味着设计规格中定义的关键功能点都已经被验证过。

## 4. 示例代码说明

配套示例代码 `01_coverage_overview.sv` 演示了：

1. 功能覆盖率与代码覆盖率的本质区别
2. 如何用 `covergroup` + `coverpoint` 定义最基本的功能覆盖率
3. 如何通过 `sample()` 对事务数据进行采样
4. 覆盖率缺口的概念——哪些 bin 未被命中说明验证不充分

核心代码片段：

```systemverilog
// 定义覆盖率组
covergroup OpCovGroup;
  coverpoint opcode {
    bins b_add = {OPCODE_ADD};
    bins b_sub = {OPCODE_SUB};
    // ... 每个操作码一个 bin
  }
endgroup

// 使用: 创建实例 → 采样
op_cov = new();
op_cov.sample();   // 对 opcode 进行采样
```

> **编译运行**：执行 `run_sv.bat Code_Examples/Chapter9_Coverage/01_coverage_overview.sv` 即可。
