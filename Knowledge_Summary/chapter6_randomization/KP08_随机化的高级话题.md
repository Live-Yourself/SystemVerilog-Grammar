# 6.8 随机化的高级话题

## 核心概念

本节探讨随机化在实际工程中的进阶议题：稳定性（reproducibility）、多对象联合随机化、调试技巧以及与覆盖率收集的结合。这些内容是将随机化从"能用"提升到"用好"的关键。

## 1. 随机化的稳定性（Reproducibility）

### 什么是稳定性

稳定性指相同的测试在相同条件下产生相同的随机序列。这是验证工程的基本要求——发现 bug 后必须能稳定复现。

### 影响稳定性的因素

| 因素 | 影响 | 控制方法 |
|------|------|---------|
| 随机种子（seed） | 不同种子产生不同序列 | `srandom()` 或 `+ntb_random_seed` |
| 对象创建顺序 | 改变对象创建顺序会改变序列 | 固定 testbench 结构 |
| 约束模式变化 | `constraint_mode`/`rand_mode` 改变有效约束集 | 在 seed 之后统一设置 |
| 仿真工具版本 | 不同版本的求解器行为可能不同 | 锁定工具版本 |
| 进程调度 | `fork...join_any` 等并发结构可能改变随机化顺序 | 尽量串行化随机化 |

### 保持稳定的最佳实践

```systemverilog
class TestEnv;
  // 1. 在 build_phase 中统一设置种子
  function void build_phase();
    process::self().srandom(SEED);
  endfunction

  // 2. 在 connect_phase 中统一设置约束模式
  function void connect_phase();
    foreach (agents[i]) begin
      agents[i].pkt.c_debug.constraint_mode(0);  // 禁用调试约束
    end
  endfunction

  // 3. 在 run_phase 中按固定顺序执行随机化
  task run_phase();
    for (int i = 0; i < num_tests; i++) begin
      agents[0].generate();  // 固定顺序
      agents[1].generate();
    end
  endtask
endclass
```

### 种子管理策略

- **调试期**：使用固定种子，每次仿真结果一致
- **回归期**：运行多轮不同种子，增加覆盖概率
- **发布期**：记录发现 bug 时的种子，存入回归配置

## 2. 多对象随机化

### 为什么需要多对象随机化

在实际验证中，一次事务可能涉及多个对象的协同随机化。例如：一个数据包的源地址必须等于另一个数据包的目的地址。

### randomize() 传参随机化多个变量

```systemverilog
class Port;
  rand bit [7:0] addr;
  rand bit [7:0] data;
endclass

Port p1, p2;
// 同时随机化两个对象的变量
p1.randomize(p1.addr, p2.addr) with {
  p1.addr + p2.addr < 200;
};
```

### std::randomize() —— 全局随机化函数

`std::randomize()` 可以随机化任意变量，不限于类成员：

```systemverilog
bit [7:0] a, b;
Port p;

// 随机化局部变量 a, b 和对象的成员 p.addr
std::randomize(a, b, p.addr) with {
  a + b < 100;
  p.addr == a;
};
```

`std::randomize()` 与 `randomize()` 的区别：

| 特性 | obj.randomize() | std::randomize() |
|------|----------------|-----------------|
| 调用方式 | 对象方法 | 系统函数 |
| 类内约束 | 自动生效 | 不生效 |
| 作用对象 | 对象的 rand 变量 | 任意变量 |
| 适用场景 | 对象内部随机化 | 跨对象/跨域随机化 |

### 对象间的约束关系

```systemverilog
class MasterTxn;
  rand bit [7:0] src_addr;
endclass

class SlaveTxn;
  rand bit [7:0] dst_addr;
endclass

MasterTxn m_txn;
SlaveTxn  s_txn;

// 要求 master 的源地址 == slave 的目的地址
std::randomize(m_txn.src_addr, s_txn.dst_addr) with {
  m_txn.src_addr == s_txn.dst_addr;
  m_txn.src_addr inside {[0:31]};
};
```

## 3. 随机化的调试技巧

### 启用随机化调试输出

大多数仿真器提供随机化调试选项：

```systemverilog
// VCS
// 编译选项: -debug_access+all
// 运行选项: +rand_debug

// Questa
// 运行选项: -randomize_debug
```

### 手动打印调试信息

```systemverilog
class DebugPacket;
  rand bit [7:0] addr;
  rand bit [7:0] data;

  function void post_randomize();
    $display("[RAND_DEBUG] addr=0x%02h, data=0x%02h at time=%0t",
             addr, data, $time);
  endfunction
endclass
```

### 定向测试辅助

在调试时，用 inline 约束将随机测试转为定向测试：

```systemverilog
// 调试：固定特定值，排除随机因素
pkt.randomize() with { addr == 8'hAB; data == 8'hCD; };

// 调试：逐步放宽约束
pkt.randomize() with { addr == 8'hAB; };  // 只固定 addr，data 随机
pkt.randomize();  // 完全随机
```

### 约束冲突定位

当随机化失败时，逐步排查方法：

```systemverilog
// 1. 先禁用所有约束，看是否成功
pkt.constraint_mode(0);
pkt.randomize();  // 如果失败，说明不是约束问题

// 2. 逐个启用约束，定位冲突
pkt.c_addr.constraint_mode(1);
pkt.randomize();  // 成功则继续

pkt.c_data.constraint_mode(1);
pkt.randomize();  // 失败则 c_addr 和 c_data 冲突
```

## 4. 随机化与覆盖率收集

### 覆盖率驱动的随机化

随机化的最终目标是**覆盖率收敛**——用尽可能少的随机事务覆盖尽可能多的功能点。

### 在 post_randomize() 中采样覆盖点

```systemverilog
class CoverageTransaction;
  rand bit [7:0] addr;
  rand bit [7:0] data;
  rand bit [1:0] op;

  // 覆盖组
  covergroup cg_trans;
    cp_addr: coverpoint addr {
      bins low    = {[0:31]};
      bins mid    = {[32:127]};
      bins high   = {[128:255]};
    }
    cp_op: coverpoint op;
    cross_addr_op: cross cp_addr, cp_op;
  endgroup

  function new();
    cg_trans = new();
  endfunction

  function void post_randomize();
    cg_trans.sample();  // 每次随机化后自动采样
  endfunction
endclass
```

### 基于覆盖率的约束调整

当某些覆盖点一直未被命中时，可以通过 inline 约束引导随机化方向：

```systemverilog
// 如果 high 地址区覆盖不足，临时引导
if (cg_trans.cp_addr.get_coverage() < 50) begin
  pkt.randomize() with { addr inside {[128:255]}; };
end else begin
  pkt.randomize();  // 正常随机
end
```

### 覆盖率收敛的注意事项

- 不要过早引导：让随机化有足够机会自然覆盖
- 种子轮换：同一测试跑多轮不同种子
- 分阶段验证：先用宽松约束快速覆盖主要功能，再用严格约束覆盖边界

## 总结

| 话题 | 核心要点 |
|------|---------|
| 稳定性 | 固定种子、固定创建顺序、固定约束模式 |
| 多对象随机化 | `std::randomize()` 跨对象联合约束 |
| 调试技巧 | 逐步禁用约束定位冲突，inline 约束转为定向测试 |
| 覆盖率结合 | post_randomize 中采样，覆盖率不足时引导约束 |
