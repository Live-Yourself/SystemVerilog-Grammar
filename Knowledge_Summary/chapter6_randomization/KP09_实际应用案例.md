# 6.9 实际应用案例

## 核心概念

前八节涵盖了 SystemVerilog 随机化的所有语法和机制，本节通过四个贴近真实工程场景的案例，展示如何将前面学到的知识**综合运用**到实际验证中。每个案例对应一种常见的验证需求：以太网数据包生成、AXI 总线事务、寄存器配置、错误注入测试。

## 1. 案例一：以太网数据包随机生成

### 场景描述

以太网 MAC 层数据包包含目的地址、源地址、类型/长度、有效载荷和 FCS 校验。需要随机生成合法的数据包，同时支持各种边界和错误场景的定向测试。

### 设计要点

- 目的/源 MAC 地址：6 字节，用动态数组随机化
- 有效载荷长度：46~1500 字节（以太网最小/最大帧长限制）
- 类型字段：区分 IPv4(0x0800)、ARP(0x0806)、VLAN(0x8100) 等协议
- FCS：在 `post_randomize()` 中计算 CRC32
- 用 `dist` 权重控制不同协议类型出现的概率

## 2. 案例二：AXI 总线事务随机化

### 场景描述

AXI 总线有 5 个独立通道（AW、W、B、AR、R），每个通道有各自的信号和控制逻辑。需要随机生成读写事务，遵守 AXI 协议规则。

### 设计要点

- 地址对齐：根据数据宽度约束地址对齐（如 32 位传输，地址 4 字节对齐）
- 突发长度：INCR（递增）、FIXED（固定）、WRAP（回绕）三种模式
- 数据与 strobe 一致性：字节选通信号与有效数据对应
- 读/写事务分离：用枚举类型 + 条件约束区分读写
- AWID/ARID 随机化：模拟多主多从场景

## 3. 案例三：寄存器配置随机化

### 场景描述

SoC 中通常有大量配置寄存器，每个寄存器有不同的有效位域、读写属性和复位值。需要随机配置寄存器组，验证各种配置组合下的 DUT 行为。

### 设计要点

- 寄存器模型：用结构体或类表示每个寄存器的位域
- 位域独立性：每个位域有自己的有效值范围
- 读写属性：只读/只写/读写属性决定是否参与随机化
- 互斥约束：某些寄存器之间有互斥关系（如模式 A 和模式 B 不能同时使能）
- 复位值：在 `pre_randomize()` 中加载复位值作为默认基线

## 4. 案例四：错误注入随机化

### 场景描述

验证 DUT 的错误处理能力时，需要在正常事务中随机注入各类错误，如校验错、协议违例、超时等。

### 设计要点

- 错误开关：用 `soft inject_err == 0` 默认不注入，需要时覆盖
- 错误类型选择：用 `dist` 权重控制不同错误类型的出现概率
- 错误位置：在长事务中随机选择出错的位置（如第几个 beat 出错）
- 严重程度：轻微错误（单 bit 翻转）vs 严重错误（全 0/全 1）
- 恢复策略：注入错误后，后续事务恢复正常，验证 DUT 的恢复能力

## 综合总结

通过这四个案例可以看出，SystemVerilog 随机化的实际应用模式：

| 案例 | 核心随机化技术 | 典型语法 |
|------|--------------|---------|
| 以太网数据包 | 动态数组 + post 计算 + 权重 | `rand dyn_array`, `post_randomize`, `dist` |
| AXI 总线 | 枚举 + 条件约束 + 对齐 | `rand enum`, `if-else`, `->` |
| 寄存器配置 | 位域 + 属性控制 + 互斥 | `rand bit field`, `constraint_mode`, `solve before` |
| 错误注入 | 软约束 + inline 覆盖 + 位置选择 | `soft`, `with`, `dist` |

这些案例的技术组合覆盖了本章所有知识点的实际用法，是构建真实测试平台的基础。
