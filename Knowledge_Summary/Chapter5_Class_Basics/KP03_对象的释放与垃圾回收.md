## 知识点3: 对象的释放与垃圾回收

### 核心概念

SystemVerilog**没有**`delete`操作符来手动释放对象。对象的内存由仿真器自动管理:当没有任何句柄指向某个对象时,该对象就成为垃圾回收的目标。程序员能做的是管理好句柄引用,决定对象"何时可以被回收",而"何时真正回收"由仿真器决定。

### 对象生命周期的三种状态

| 状态 | 说明 | 句柄值 |
|------|------|--------|
| **未创建** | 仅声明了句柄,未分配内存 | `null` |
| **活跃** | 至少有一个句柄引用对象 | 有效地址 |
| **可回收** | 所有句柄都不再引用该对象 | 无引用 |

### 句柄置null释放对象

将句柄赋值为`null`是主动断开句柄与对象关联的标准方式:

```systemverilog
Transaction tr;
tr = new();   // 创建对象, tr指向该对象
// ... 使用对象 ...
tr = null;    // 断开引用, 无其他句柄引用时可被回收
```

### 多句柄引用下的垃圾回收

**关键规则**: 只有当指向对象的**所有句柄**都不再引用该对象时,该对象才会被垃圾回收。

```systemverilog
Transaction tr1, tr2;
tr1 = new();      // 创建对象, tr1引用它
tr2 = tr1;        // tr2也引用同一对象(句柄赋值,非对象复制)

tr1 = null;       // 仅断开tr1, tr2仍引用 → 对象不会被回收!
tr2.display();    // 通过tr2仍可正常访问

tr2 = null;       // 所有引用断开 → 对象可被垃圾回收
```

用`== null`判断句柄是否为空: `(tr1 == null)` 返回真表示句柄未指向对象。

### 句柄重指向 -- 隐式释放

当句柄被赋值为一个新对象时,句柄自动断开与旧对象的引用:

```systemverilog
tr = new(...);    // tr → 对象A
tr = new(...);    // tr → 对象B, 对象A不再被引用, 可被回收
```

不需要显式`tr = null`,重指向本身就断开了旧引用。

### 垃圾回收的时机

SystemVerilog标准并未规定垃圾回收的具体执行时机。仿真器可选择在空闲时、内存不足时、或仿真结束时执行。程序员**无法显式控制**回收时机。

### 引用计数概念

虽然标准不强制引用计数实现,但这个概念有助于理解回收条件:

```
new()创建    → 引用计数 = 1
句柄赋值     → 引用计数 +1
句柄置null   → 引用计数 -1
句柄重指向   → 旧对象引用计数 -1
引用计数 = 0 → 对象可被垃圾回收
```

**引用计数归零是对象可被回收的充要条件。**

### 局部句柄的作用域

`begin/end`块中声明的句柄是局部变量,离开块后句柄失效:

```systemverilog
begin
  Transaction local_tr = new(...);
  local_tr.display();  // 块内正常使用
end
// local_tr已离开作用域,不可访问
// 若该对象无其他引用,也可被回收
```

### 动态数组与对象的内存管理

动态数组也是自动管理的,句柄被重新赋值或置null时,原数组空间也被回收:

```systemverilog
class Packet;
  bit [7:0] payload[];
  function new(int size);
    payload = new[size];
  endfunction
endclass

Packet pkt = new(16);
pkt = null;  // pkt及其内部的payload数组都可被回收
```

### 验证实践中的内存管理

队列、动态数组等容器持有对象句柄时会阻止回收。长时间仿真中需定期清理:

```systemverilog
class Generator;
  Transaction queue[$];
  function void run();
    for (int i = 0; i < N; i++) begin
      Transaction t = new();
      queue.push_back(t);  // 引用被队列持有,不会回收
    end
    queue.delete();  // 清理: 释放所有句柄引用, 对象可被回收
  endfunction
endclass
```

### 关键要点

- SystemVerilog采用**自动垃圾回收**,没有`delete`操作符
- 只有**所有句柄**都不再引用对象时,对象才可被回收
- 句柄置`null`是主动断开引用的标准方式
- 垃圾回收的**时机不可控**,由仿真器决定
- 队列/数组持有句柄会阻止回收,长期仿真需定期清理

### 典型应用场景

- **事务处理**: Driver发送完事务后将句柄置null
- **响应队列**: Scoreboard比对完成后清理已完成的事务对象
- **配置切换**: 切换测试配置时释放旧配置对象
- **大规模回归**: 避免无限制的对象积累导致内存溢出
