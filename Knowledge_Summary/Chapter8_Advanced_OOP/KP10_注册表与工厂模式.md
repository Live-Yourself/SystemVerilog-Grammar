# KP10 测试注册表与工厂模式（Registry & Factory Pattern）

## 核心概念

### 工厂模式

工厂模式的本质是：**把 `new()` 的权利集中到一个"工厂"里，由工厂根据需求创建对应类型的对象。** 就像你买手机不需要知道手机是哪个工厂流水线生产的，只需要告诉售货员"我要 iPhone"或"我要华为"——售货员（工厂）负责找到对应的工厂去生产。

**解决的问题**：调用方不需要 `import` 具体子类，也不需要知道子类的类名。只需要给一个"类型标识"，工厂就帮你创建正确的子类对象。

### 测试注册表

注册表是一个全局的"字典"，用字符串 key 映射到对应的类。工厂创建对象时，通过查注册表来找到应该创建哪个类。

**两者结合**：注册表负责"登记"（哪个字符串对应哪个类），工厂负责"生产"（根据字符串查注册表并创建对象）。

---

## 1. 工厂模式的基本结构

```systemverilog
class Factory;
    static function Transaction create(string type_name);
        if (type_name == "read")
            return new("read_txn");
        else if (type_name == "write")
            return new("write_txn");
        else
            return null;
    endfunction
endclass

Transaction txn = Factory::create("read");  // 不需要知道具体类名
```

---

## 2. 用注册表替代 if-else

当子类很多时，if-else 会很长。注册表用一个关联数组替代：

```systemverilog
class Factory;
    typedef Transaction (*creator_t)(string);  // 函数指针类型
    static creator_t registry[string];          // 关联数组：字符串 → 创建函数

    static function void register(string name, creator_t creator);
        registry[name] = creator;
    endfunction

    static function Transaction create(string name);
        if (registry.exists(name))
            return registry[name]("auto");
        else begin
            $error("未注册的类型: %0s", name);
            return null;
        end
    endfunction
endclass
```

---

## 3. 工厂模式的价值

| 对比 | 直接 new | 工厂模式 |
|------|---------|---------|
| 调用方是否知道具体类 | 是 | 否，只知道字符串标识 |
| 新增子类 | 所有调用处都要改 | 只需注册，调用处不改 |
| 运行时动态选择类型 | 不能 | 可以（根据 test 参数选择） |
| 可替换性 | 低（硬编码） | 高（改配置即可替换） |

**这是 UVM 中 `uvm_factory` 和 `uvm_object_registry` 的核心原理。**
