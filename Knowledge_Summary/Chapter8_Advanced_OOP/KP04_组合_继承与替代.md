# KP04 组合、继承和替代（Composition, Inheritance, and Delegation）

## 核心概念

在面向对象设计中，当你需要在类 A 中使用类 B 的功能时，有三种基本方式。选择哪种方式，是设计验证平台时的常见决策。

---

## 三种方式的对比

### 1. 继承（Inheritance）—— "是一种"

```systemverilog
class ReadWriteTxn extends Transaction;
    // 自动拥有 Transaction 的所有成员
endclass
```

含义：`ReadWriteTxn` **是一种** `Transaction`。它继承了父类的全部功能，还可以覆盖和扩展。

### 2. 组合（Composition）—— "有一个"

```systemverilog
class Driver;
    Transaction txn;   // Driver 中包含一个 Transaction 对象
endclass
```

含义：`Driver` **有一个** `Transaction`。它不继承 Transaction 的任何东西，只是把 Transaction 作为自己的一个成员来使用。

### 3. 替代/委托（Delegation）—— "借用功能"

```systemverilog
class BusTxn;
    Transaction low_level_txn;   // 内部持有一个 Transaction 对象
    function void set_addr(int a);
        low_level_txn.addr = a;  // 把调用转发给内部对象
    endfunction
endclass
```

含义：`BusTxn` 不继承也不直接暴露 `Transaction`，而是把功能调用**委托/转发**给内部的对象。外部看起来 `BusTxn` 自己有这些功能，但实际干活的是内部的 `Transaction`。

---

## 如何选择

| 场景 | 推荐方式 | 理由 |
|------|:-------:|------|
| 子类是父类的特殊版本（is-a 关系） | 继承 | 如 ReadTxn、WriteTxn 是 Transaction 的子类型 |
| 类之间是"整体-部分"关系（has-a 关系） | 组合 | 如 Driver 拥有一个 Transaction，但 Driver 不是 Transaction |
| 想复用功能但不想暴露内部实现 | 委托 | 如 BusTxn 借用 Transaction 的字段，但有自己的接口 |

**经验法则**：如果"是一种"就用继承，如果"有一个"就用组合。验证平台中组合比继承更常见——组件之间更多的是"包含"关系而非"父子"关系。
